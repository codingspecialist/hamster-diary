2026-04-09 Kubernetes

비유로 보는 Kubernetes

Kubernetes 클러스터를 회사 건물에 비유하면 모든 컴포넌트가 직관적으로 이해된다.

K8s 컴포넌트비유핵심 역할
Cluster회사 건물 전체모든 자원을 포함하는 최상위 단위
Node사무실 층실제 컴퓨팅 자원 (CPU, 메모리)
Pod직원 (내선번호 = Pod IP)앱이 실행되는 최소 단위, 죽으면 새 직원이 옴 (IP 바뀜)
Deployment인사팀직원(Pod) 수 관리, 퇴사하면 자동 충원, 버전 교체
Service (ClusterIP)대표번호직원이 바뀌어도 대표번호는 그대로, 내부 통화 전용
kube-proxy교환기 (PBX)대표번호로 온 전화를 현재 근무 중인 직원에게 연결 (iptables DNAT)
CoreDNS내선번호부이름("backend-service") → 번호(10.96.x.x) 변환
ConfigMap공용 게시판환경설정 공유 (DB 주소, 포트 등). 누구나 볼 수 있음
Secret금고비밀번호, 인증서 보관. base64 인코딩 (암호화는 아님)
PV/PVC공용 캐비닛직원이 퇴사해도 서류(데이터)는 캐비닛에 남아있음
Namespace부서같은 건물이지만 부서별 자원 격리 (metacoding, default 등)
NodePort건물 외벽 전용 출입구외부에서 특정 포트로 직접 들어옴 (30000-32767)
LoadBalancer건물 안내데스크 (외주)클라우드 업체가 운영하는 외부 안내데스크, 방문객을 적절한 층으로 안내
Ingress건물 정문 안내원URL 경로별로 다른 부서로 안내 (/api → backend, / → frontend)

Service 타입별 접근 범위 — "누가 전화할 수 있는가?"

Service 타입은 "이 대표번호로 누가 전화할 수 있는가?"를 결정한다:

+--------------------------------------------------------+
|                                                         |
|  ClusterIP (내선전화)                                    |
|  -------------------------------------------------------+
|  OK: 클러스터 내부 Pod끼리만 통화 가능                     |
|  NG: 외부에서 전화 불가                                   |
|  -> "사내 내선전화. 외부에서 이 번호로 전화하면 안 됨"      |
|                                                         |
|  +---------------------------------------------------+  |
|  |                                                    |  |
|  |  NodePort (외벽 출입구)                              |  |
|  |  --------------------------------------------------+  |
|  |  OK: ClusterIP 기능 포함 (내부 통화 가능)            |  |
|  |  OK: + Node IP:30080으로 외부 접근 가능              |  |
|  |  -> "건물 외벽에 전용 출입구를 뚫은 것"               |  |
|  |                                                    |  |
|  |  +----------------------------------------------+  |  |
|  |  |                                               |  |  |
|  |  |  LoadBalancer (외주 안내데스크)                  |  |  |
|  |  |  ---------------------------------------------+  |  |
|  |  |  OK: NodePort 기능 포함                        |  |  |
|  |  |  OK: + 클라우드가 외부 IP(52.14.x.x) 할당      |  |  |
|  |  |  -> "클라우드 업체가 안내데스크를 세워줌"         |  |  |

포함 관계: LoadBalancer 포함 NodePort 포함 ClusterIP

왜 ClusterIP만으로는 외부 접근이 안 되는가?

외부 (192.168.0.x) -> ClusterIP (10.96.x.x) ← 불가능!

이유:
  10.96.x.x는 Node 내부의 iptables에만 존재하는 가상 IP
  외부 네트워크에서는 이 IP 대역으로 가는 경로(route)가 없음
  -> Node까지 도달해야 kube-proxy가 변환해줄 수 있음

해결:
  NodePort = Node IP(192.168.49.2):30080으로 들어오게 해서
             kube-proxy가 ClusterIP -> Pod로 전달
  LoadBalancer = 클라우드가 외부 IP를 붙여서 NodePort로 연결

Service 타입 정리

타입접근 범위누가 만드는가비유언제 쓰는가
ClusterIP내부만kube-proxy (iptables)내선전화DB, Redis 등 내부 서비스
NodePort내부 + 외부kube-proxy (iptables)외벽 출입구개발/테스트 환경 외부 접근
LoadBalancer내부 + 외부 + 외부IP클라우드(AWS/GCP) + kube-proxy외주 안내데스크프로덕션 외부 노출
ExternalName내부→외부CoreDNS (CNAME)외부 업체 전화번호 별칭외부 DB/API 추상화
Headless내부 (Pod 직접)CoreDNS (A 레코드)직원 내선번호 직접 조회StatefulSet (DB 리더/팔로워)

외부 접근 3가지 방법

1. NodePort — 건물 외벽에 출입구 뚫기

[브라우저] -> 192.168.49.2:30080
                    |
             kube-proxy (iptables)
                    |
             ClusterIP -> Pod
  • Node IP + 포트(30000-32767)로 직접 접근
  • 포트 번호가 제한적이고, Node IP를 알아야 함
  • 개발/테스트 환경에 적합

2. kubectl port-forward — 직원에게 직통전화

[브라우저] -> localhost:3000
                    |
             kubectl이 터널 생성
                    |
             Pod 10.244.0.5:3000 (직접 연결)
  • Service/kube-proxy를 거치지 않고 Pod에 직접 연결
  • kubectl 프로세스가 살아있어야 함
  • 디버깅/개발 용도 (일시적)

3. minikube tunnel — LoadBalancer 시뮬레이션

[브라우저] -> 127.0.0.1:80 (또는 할당된 External IP)
                    |
             minikube tunnel (별도 프로세스)
                    |
             LoadBalancer Service -> NodePort -> kube-proxy -> Pod
  • 클라우드 없이 LoadBalancer 타입을 로컬에서 테스트
  • minikube tunnel 프로세스가 실행 중이어야 함
  • 프로덕션 배포 전 테스트 용도

실제 요청 흐름 시나리오

사용자가 브라우저에서 프론트엔드를 열고, 백엔드 API를 호출하는 전체 과정:

1. 브라우저 -> NodePort(192.168.49.2:30080)
   "외벽 출입구로 들어옴"

2. kube-proxy가 iptables로 frontend-service(10.96.0.50:80)를 찾음
   "교환기가 대표번호를 확인"

3. kube-proxy가 frontend Pod 중 하나(10.244.0.3:80)로 DNAT
   "교환기가 현재 근무 중인 직원에게 연결"

4. frontend Pod(Nginx)가 HTML/JS 반환
   "직원이 응답"

5. 브라우저의 JS가 backend-service:8080으로 API 요청
   -> frontend Pod 내부에서 CoreDNS 조회
   "내선번호부에서 backend-service 번호를 찾음 -> 10.96.0.100"

6. kube-proxy가 backend Pod 중 하나(10.244.0.5:8080)로 DNAT
   "교환기가 백엔드 직원에게 연결"

7. backend Pod가 ConfigMap에서 DB 주소 읽음
   "공용 게시판에서 DB 연결 정보 확인"
   DB_HOST=db-service -> CoreDNS -> 10.96.0.200

8. backend Pod가 Secret에서 DB 비밀번호 읽음
   "금고에서 비밀번호 꺼냄"

9. kube-proxy가 db-service(10.96.0.200) -> MySQL Pod(10.244.0.7)로 연결
   "교환기가 DB 직원에게 연결"

10. MySQL Pod가 PV(영구 볼륨)에서 데이터 조회
    "직원이 공용 캐비닛에서 서류를 찾음"

11. 응답이 역순으로 돌아감
    MySQL -> backend -> frontend -> 브라우저

컴포넌트 관계 요약

Pod가 시작할 때 읽는 것:
  +-- ConfigMap -> 환경변수 주입 (DB_HOST, REDIS_PORT 등)
  +-- Secret -> 비밀 환경변수 주입 (DB_PASSWORD 등)
  +-- PVC -> 볼륨 마운트 (데이터 저장 경로)

Pod가 다른 Pod에 접근할 때:
  +-- CoreDNS에 Service 이름 질의 -> ClusterIP 반환
  +-- kube-proxy가 ClusterIP -> Pod IP로 DNAT
  +-- 로드밸런싱으로 여러 Pod 중 하나에 연결

외부에서 Pod에 접근할 때:
  +-- NodePort: Node IP:포트 -> kube-proxy -> Pod
  +-- LoadBalancer: 외부 IP -> NodePort -> kube-proxy -> Pod
  +-- port-forward: localhost:포트 -> kubectl 터널 -> Pod (직접)
