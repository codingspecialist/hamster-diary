Kafka + Python + Docker 비동기 메시징 예제

2026-04-13 Kafka
DB 없이 Docker로 Kafka를 띄우고, Python Producer/Consumer 두 개를 붙여 "비동기 메시지 큐"가 어떻게 동작하는지 눈으로 확인하는 학습용 예제.

1. Kafka가 뭔가요? — 우체국 비유

Kafka를 한 줄로 표현하면 "메시지 우체국"입니다.

   ┌────────────┐    편지를 부침     ┌──────────────┐    편지를 가져감   ┌────────────┐
   │  Producer  │ ─────────────────▶ │    Kafka     │ ─────────────────▶ │  Consumer  │
   │  (보내는 쪽)│                    │   (우체국)    │                    │ (받는 쪽)   │
   └────────────┘                    └──────────────┘                    └────────────┘
                                       ▲    │
                                       │    └─ 편지를 보관함 (디스크에 저장)
                                       └─ 받는 사람이 없어도 OK
  • Producer: 편지를 우체국에 던지고 바로 자기 일 하러 감 → "비동기"
  • Kafka: 편지를 안전하게 보관 (받는 사람이 늦게 와도 안 사라짐)
  • Consumer: 자기 페이스대로 편지함을 비움

이게 DB와 가장 큰 차이입니다. DB는 "조회"하지만 Kafka는 "흐름"입니다.

Topic = 우편함 종류

한 우체국 안에 여러 우편함이 있다고 생각하면 됩니다.

  Kafka 우체국
  ├─ 📬 topic: "orders"      (주문 메시지함)
  ├─ 📬 topic: "payments"    (결제 메시지함)
  └─ 📬 topic: "demo"        (이번 예제에서 사용)

2. 왜 비동기인가? — 카페 비유

직접 비교해봅시다.

동기 (HTTP 직접 호출) — 카페에서 줄 서기

손님 ──(주문)──▶ 바리스타 ──(커피 만드는 동안 손님 대기)──▶ 손님 받음
                  ⏱ 손님은 5분간 아무것도 못함

비동기 (Kafka) — 카페에서 진동벨 받기

손님 ──(주문 던짐)──▶ 진동벨 받고 바로 자리로 ✅
                          │
                          ▼
                   바리스타가 천천히 만들고
                   완성되면 진동벨이 울림 🔔

→ Producer는 "주문만 던지고" 자기 할 일을 합니다.
→ Consumer는 "큐에 쌓인 일"을 자기 속도로 처리합니다.

5. 코드 — 한 줄씩 의미와 함께

5-1. docker-compose.yml

services:
  kafka:
    image: bitnami/kafka:3.7
    ports:
      - "9092:9092"
    environment:
      # KRaft 모드: Zookeeper 없이 혼자 동작
      KAFKA_CFG_NODE_ID: 0
      KAFKA_CFG_PROCESS_ROLES: controller,broker
      KAFKA_CFG_CONTROLLER_QUORUM_VOTERS: 0@kafka:9094
      KAFKA_CFG_LISTENERS: PLAINTEXT://:9093,CONTROLLER://:9094,EXTERNAL://:9092
      KAFKA_CFG_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9093,EXTERNAL://localhost:9092
      KAFKA_CFG_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT,EXTERNAL:PLAINTEXT
      KAFKA_CFG_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_CFG_INTER_BROKER_LISTENER_NAME: PLAINTEXT
      KAFKA_CFG_AUTO_CREATE_TOPICS_ENABLE: "true"   # topic 자동 생성

  producer:
    build: ./producer
    ports:
      - "8000:8000"
    environment:
      KAFKA_BOOTSTRAP: kafka:9093    # 컨테이너끼리는 내부 주소
    depends_on:
      - kafka

  consumer:
    build: ./consumer
    environment:
      KAFKA_BOOTSTRAP: kafka:9093
    depends_on:
      - kafka
🔑 리스너가 두 개인 이유: 컨테이너끼리 통신 (kafka:9093)과 내 PC에서 접근 (localhost:9092)을 분리해야 합니다. Kafka 처음 쓸 때 가장 헷갈리는 부분!

5-2. producer/main.py (FastAPI)

import json, os, time
from fastapi import FastAPI
from pydantic import BaseModel
from kafka import KafkaProducer
from kafka.errors import NoBrokersAvailable

app = FastAPI()

# Kafka가 부팅될 때까지 재시도 (처음 10~20초 걸림)
def make_producer():
    while True:
        try:
            return KafkaProducer(
                bootstrap_servers=os.getenv("KAFKA_BOOTSTRAP"),
                value_serializer=lambda v: json.dumps(v).encode(),
            )
        except NoBrokersAvailable:
            print("Kafka 아직 준비 안 됨, 3초 후 재시도...")
            time.sleep(3)

producer = make_producer()

class Msg(BaseModel):
    message: str

@app.post("/publish")
def publish(msg: Msg):
    producer.send("demo", {"message": msg.message, "ts": time.time()})
    # send()는 큐에 넣을 뿐, 실제 전송은 백그라운드 → 즉시 반환됨
    return {"status": "queued", "message": msg.message}

@app.get("/health")
def health():
    return {"ok": True}

핵심:

producer.send(...)  ──▶  내부 버퍼에 쌓고 즉시 return
                              │
                              └─ 백그라운드 스레드가 Kafka로 flush

5-3. consumer/main.py (워커)

import json, os, time
from kafka import KafkaConsumer
from kafka.errors import NoBrokersAvailable

def make_consumer():
    while True:
        try:
            return KafkaConsumer(
                "demo",
                bootstrap_servers=os.getenv("KAFKA_BOOTSTRAP"),
                group_id="demo-group",
                auto_offset_reset="earliest",   # 처음부터 읽기
                value_deserializer=lambda v: json.loads(v.decode()),
            )
        except NoBrokersAvailable:
            print("Kafka 아직 준비 안 됨, 3초 후 재시도...")
            time.sleep(3)

consumer = make_consumer()
print("Consumer 준비 완료, 메시지 대기 중...")

for record in consumer:
    print(f"[{time.strftime('%H:%M:%S')}] 수신: {record.value}")
    time.sleep(2)   # 일부러 천천히 처리 → 비동기 효과 확인용
    print(f"[{time.strftime('%H:%M:%S')}] 처리 완료")
🐢 일부러 sleep(2)를 넣은 이유: Producer는 0.01초만에 응답하지만 Consumer는 천천히 소화하는 모습을 로그로 확인하기 위함.

docker compose up -d --build

# 2. Consumer 로그 보기 (별도 터미널)
docker compose logs -f consumer

# 3. 메시지 5번 빠르게 던지기
for i in 1 2 3 4 5; do

    -H "Content-Type: application/json" \
    -d "{\"message\":\"hello-$i\"}"
  echo
done

7. 더 해볼 만한 실험

실험 A: Consumer를 끄고 메시지를 쌓아두기

docker compose stop consumer
# 메시지 10번 던지기
for i in $(seq 1 10); do curl -X POST ... -d "{\"message\":\"$i\"}"; done
docker compose start consumer
docker compose logs -f consumer

→ Consumer가 꺼져있어도 메시지는 Kafka에 보관됨. 다시 켜면 밀린 것부터 순서대로 처리. ✨

실험 B: Consumer를 2개로 늘리기 (consumer group)

docker compose up -d --scale consumer=2 로 띄우면 같은 group에 속한 두 워커가 메시지를 나눠서 처리합니다 (단, topic이 1 partition이면 1명만 일함).

8. 정리

개념한 줄 요약
Producer편지 던지고 바로 가는 사람
Consumer편지함 비우는 사람 (자기 페이스)
Topic우편함 종류
Broker우체국 건물 자체
Group같은 일을 나눠 하는 팀
KRaftZookeeper 없는 신형 Kafka 운영 모드
"Producer는 던진다. Consumer는 자기 속도로 먹는다. Kafka는 그 사이에서 안 잃어버린다."

이 한 문장이 비동기 메시징의 전부입니다. 🎯
