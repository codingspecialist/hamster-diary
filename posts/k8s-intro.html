1. 쿠버네티스(Kubernetes)란?

쿠버네티스(K8s)는 컨테이너 오케스트레이션 플랫폼이다. 여러 개의 컨테이너를 자동으로 배포, 확장, 관리해주는 시스템이다.

Docker Compose vs Kubernetes

항목Docker ComposeKubernetes
목적단일 서버에서 멀티 컨테이너 실행여러 서버(클러스터)에서 컨테이너 관리
규모개발/테스트 환경프로덕션 대규모 환경
자동 복구없음 (컨테이너 죽으면 수동 재시작)자동 복구 (Pod가 죽으면 자동 재생성)
스케일링수동 (scale 명령)자동/수동 스케일링 지원
로드밸런싱직접 설정 (Nginx 등)Service가 자동으로 로드밸런싱
롤링 업데이트미지원무중단 배포 지원

한 줄 요약: Docker Compose는 "한 대의 컴퓨터에서 여러 컨테이너 실행", K8s는 "여러 대의 컴퓨터에서 컨테이너를 자동 관리"

2. 핵심 개념

Cluster와 Node

┌─────────────────────────────────────────────┐
│                  Cluster                     │
│  ┌───────────────┐  ┌───────────────┐       │
│  │  Master Node  │  │  Worker Node  │       │
│  │  (제어 담당)   │  │  (실행 담당)   │       │
│  │  - API Server │  │  - Pod 실행    │       │
│  │  - Scheduler  │  │  - 컨테이너 관리│       │
│  └───────────────┘  └───────────────┘       │
└─────────────────────────────────────────────┘
  • Cluster: 쿠버네티스가 관리하는 서버들의 집합
  • Master Node: 클러스터를 제어하는 노드 (명령을 받고 스케줄링)
  • Worker Node: 실제 컨테이너(Pod)가 실행되는 노드

Pod

  • 쿠버네티스의 최소 배포 단위
  • 하나 이상의 컨테이너를 포함
  • 같은 Pod 안의 컨테이너는 네트워크와 저장소를 공유
  • Docker의 컨테이너 1개 = K8s의 Pod 1개라고 이해하면 쉽다

Deployment

  • Pod를 선언적으로 관리하는 리소스
  • replicas: 2로 설정하면 항상 2개의 Pod를 유지
  • Pod가 죽으면 자동으로 새 Pod를 생성 (자동 복구)
  • 롤링 업데이트로 무중단 배포 가능
# 예시: backend-deploy.yml
spec:
  replicas: 2        # 항상 2개의 Pod 유지
  selector:
    matchLabels:
      app: backend   # 이 라벨을 가진 Pod를 관리

Service

  • Pod에 접근하기 위한 고정 네트워크 엔드포인트
  • Pod는 생성/삭제될 때마다 IP가 바뀌지만, Service는 고정된 이름으로 접근 가능
  • backend-service:8080처럼 이름으로 다른 Pod에서 접근
# 예시: backend-service.yml
spec:
  selector:
    app: backend     # app=backend 라벨의 Pod들에게 트래픽 전달
  ports:
    - port: 8080     # Service 포트
      targetPort: 8080  # Pod 컨테이너 포트

Namespace

  • 리소스를 논리적으로 격리하는 가상 클러스터
  • 같은 클러스터 안에서 팀별, 환경별로 구분 가능
  • 이 실습에서는 metacoding 네임스페이스를 사용

Linux Namespace → Docker → Kubernetes Namespace의 관계

"Namespace"라는 이름이 같지만, 각 계층에서 격리하는 대상이 다르다. 핵심은 리눅스 커널의 Namespace 기술이 모든 것의 출발점이라는 것이다.

┌─────────────────────────────────────────────────────────────────────┐
│                        Linux Kernel                                 │
│                                                                     │
│  Namespace (커널 기능) ── OS 자원을 격리하는 기술                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │   PID    │ │   NET    │ │   MNT    │ │   UTS    │ │   IPC    │  │
│  │ 프로세스  │ │ 네트워크  │ │ 파일시스템│ │ 호스트명  │ │ 프로세스  │  │
│  │  격리    │ │  격리    │ │  격리    │ │  격리    │ │ 통신 격리 │  │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘  │
│       │                                                             │
│       ▼                                                             │
│  ┌─────────────────────────────────────────────┐                    │
│  │            Docker가 이 기술을 사용             │                    │
│  │                                             │                    │
│  │  컨테이너 A (PID/NET/MNT 격리)               │                    │
│  │  컨테이너 B (PID/NET/MNT 격리)  ← OS 수준 격리│                    │
│  │  컨테이너 C (PID/NET/MNT 격리)               │                    │
│  └─────────────────────────────────────────────┘                    │
│       │                                                             │
│       ▼                                                             │
│  ┌─────────────────────────────────────────────┐                    │
│  │       Kubernetes Namespace                   │                    │
│  │                                             │                    │
│  │  NS: dev    (Pod A, Service A)              │                    │
│  │  NS: prod   (Pod B, Service B)  ← 논리적 격리│                    │
│  │  NS: test   (Pod C, Service C)              │                    │
│  └─────────────────────────────────────────────┘                    │
└─────────────────────────────────────────────────────────────────────┘

3가지 Namespace 비교

항목Linux NamespaceDocker (Linux NS 활용)Kubernetes Namespace
격리 대상OS 자원 (PID, 네트워크, 파일시스템)컨테이너별 OS 환경K8s 리소스 (Pod, Service 등)
격리 수준커널 수준 (진짜 격리)OS 수준 (프로세스 격리)논리적 (이름 구분)
목적하나의 OS에서 독립된 환경 생성앱마다 독립된 실행 환경 제공클러스터 안에서 팀/환경별 구분
보안 격리강함 (커널이 차단)강함 (Linux NS + cgroups)약함 (네트워크 정책 별도 설정 필요)

예시로 이해하기

[ 하나의 리눅스 서버 ]

# ── Linux Namespace ──
# PID Namespace: 컨테이너 안에서 ps 치면 자기 프로세스만 보인다
# NET Namespace: 컨테이너마다 독립된 IP, 포트를 가진다
# MNT Namespace: 컨테이너마다 독립된 파일시스템을 가진다

# ── Docker가 Linux Namespace를 조합해서 컨테이너를 만든다 ──
docker run nginx    → Linux의 PID + NET + MNT Namespace 조합으로 격리된 환경 생성
docker run mysql    → 또 다른 Namespace 조합 → nginx와 완전히 격리됨

# ── Kubernetes Namespace는 "이름표"로 구분하는 것 ──
kubectl create ns dev     → dev 이름표가 붙은 공간
kubectl create ns prod    → prod 이름표가 붙은 공간
# 같은 이름의 Service를 각 Namespace에 만들 수 있다
# dev/backend-service와 prod/backend-service는 서로 다른 리소스

핵심 차이: Linux Namespace는 "벽을 세워서 진짜 못 보게" 하는 것이고, K8s Namespace는 "폴더를 나눠서 정리" 하는 것이다. Docker는 Linux Namespace를 사용해서 컨테이너라는 격리 환경을 만들고, K8s는 그 컨테이너(Pod)들을 논리적으로 그룹핑하는 데 Namespace를 사용한다.

ConfigMap과 Secret

  • ConfigMap: 민감하지 않은 설정값 저장 (DB URL, 포트 등)
  • Secret: 민감한 정보 저장 (비밀번호, 인증키 등)
  • 환경변수로 Pod에 주입하여 이미지와 설정을 분리
# ConfigMap 예시 - 일반 설정
data:
  SPRING_DATASOURCE_URL: jdbc:mysql://db-service:3306/metadb
  SPRING_DATA_REDIS_HOST: redis-service

# Secret 예시 - 민감한 정보
stringData:
  SPRING_DATASOURCE_USERNAME: metacoding
  SPRING_DATASOURCE_PASSWORD: metacoding1234

PersistentVolume (PV)와 PersistentVolumeClaim (PVC)

  • Pod는 삭제되면 내부 데이터도 사라진다
  • PV: 클러스터에 미리 만들어둔 저장 공간 (관리자가 생성)
  • PVC: Pod가 저장 공간을 요청하는 것 (개발자가 생성)
  • DB처럼 데이터가 유지되어야 하는 서비스에 필수
PV (1Gi 저장소 생성) ←── PVC (1Gi 요청) ←── Pod (마운트하여 사용)

kubectl

  • 쿠버네티스 클러스터를 제어하는 CLI 도구
  • kubectl apply, kubectl get, kubectl logs 등의 명령어로 리소스를 관리

3. Minikube란?

  • 로컬 환경에서 싱글 노드 쿠버네티스 클러스터를 실행하는 도구
  • 실제 프로덕션에서는 여러 Node를 사용하지만, 학습/테스트 목적으로 1대의 컴퓨터에서 K8s를 체험
  • Master Node와 Worker Node가 하나의 VM(또는 컨테이너) 안에서 동작

노드(Node)란?

노드는 쿠버네티스가 Pod를 실행하는 서버(컴퓨터) 1대를 말한다. 물리 서버든 VM이든 상관없다.

노드 종류역할비유
Master Node (Control Plane)클러스터 전체를 관리, 명령을 받고 스케줄링식당 매니저 — 주문 접수, 요리사 배치, 상황 판단
Worker Node실제 Pod(컨테이너)를 실행하는 곳식당 요리사 — 실제 요리(앱)를 만드는 사람
 
Master Node가 하는 일:
  ├── API Server     → kubectl 명령을 받는 창구
  ├── Scheduler      → "이 Pod는 어느 Worker에서 실행할까?" 결정
  ├── Controller     → "replicas: 2인데 1개 죽었네? 1개 더 만들어!" 자동 복구
  └── etcd           → 클러스터 상태를 저장하는 DB

Worker Node가 하는 일:
  ├── kubelet        → Master 명령을 받아서 Pod를 실행/관리
  ├── Container Runtime (Docker 등) → 실제 컨테이너를 실행
  └── kube-proxy     → Pod 간 네트워크 통신 처리

왜 여러 노드를 사용하는가?

이유 1: 고가용성 (서버가 죽어도 서비스 유지)

Worker Node 1개만 있을 때 (Minikube):
  Worker Node 1 [Pod A, Pod B, Pod C]
       ↓ 서버 고장!
  전체 서비스 중단

Worker Node 여러 개일 때 (프로덕션):
  Worker Node 1 [Pod A, Pod B]  ← 이 서버가 고장나도
  Worker Node 2 [Pod A, Pod C]  ← 여기서 계속 서비스!
  Worker Node 3 [Pod B, Pod C]  ← 여기서도 계속 서비스!
  → K8s가 자동으로 죽은 Pod를 다른 Node에 재생성

이유 2: 부하 분산 (트래픽을 나눠서 처리)

사용자 1000명이 동시 접속:

Node 1개: 한 서버가 전부 처리 → 서버 과부하 → 느려짐
Node 3개: 각 서버가 ~333명씩 처리 → 쾌적

쉬운 예시:
  식당에 요리사 1명 → 주문 100개 밀림 (Minikube)
  식당에 요리사 5명 → 주문 100개를 나눠서 처리 (프로덕션)

이유 3: 장애 격리 (문제가 퍼지지 않음)

Node 1에서 메모리 누수 발생:
  Node 1 [Backend Pod - 메모리 폭주]  ← 이 Node만 영향
  Node 2 [DB Pod - 정상]             ← 다른 Node는 무관
  Node 3 [Frontend Pod - 정상]       ← 다른 Node는 무관

1개 Node였다면?
  Node 1 [Backend + DB + Frontend]  ← 전부 죽음

싱글 노드 vs 멀티 노드 아키텍처 비교

Minikube (싱글 노드) — 학습용

┌─────────────────────────────────────────────┐
│           내 컴퓨터 (1대)                     │
│                                             │
│  ┌────────────────────────────────────────┐ │
│  │   Minikube VM (Master + Worker 합체)   │ │
│  │                                        │ │
│  │   [Master 역할]  [Worker 역할]          │ │
│  │   API Server     Pod: Frontend         │ │
│  │   Scheduler      Pod: Backend x2       │ │
│  │   Controller     Pod: DB               │ │
│  │   etcd           Pod: Redis            │ │
│  │                                        │ │
│  │   → 전부 한 곳에서 실행 (장애 = 전체 중단)│ │
│  └────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘

프로덕션 (멀티 노드) — 실서비스용

┌──────────────────────────────────────────────────────────────────────┐
│                        K8s 클러스터                                   │
│                                                                      │
│  ┌──────────────────┐                                                │
│  │  Master Node     │  ← 관리만 담당 (Pod 실행 안 함)                  │
│  │  API Server      │                                                │
│  │  Scheduler       │  "Backend Pod 1개 죽었네?                       │
│  │  Controller      │   Worker Node 3에 새로 만들어!"                  │
│  │  etcd            │                                                │
│  └────────┬─────────┘                                                │
│           │ 명령 전달                                                  │
│     ┌─────┼──────────────┬───────────────┐                           │
│     ▼     ▼              ▼               ▼                           │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐                  │
│  │ Worker Node 1│ │ Worker Node 2│ │ Worker Node 3│                  │
│  │ (서버 1대)    │ │ (서버 1대)    │ │ (서버 1대)    │                  │
│  │              │ │              │ │              │                  │
│  │ Pod:Frontend │ │ Pod:Backend  │ │ Pod:Backend  │                  │
│  │ Pod:Redis    │ │ Pod:DB       │ │ (자동 복구용)  │                  │
│  └──────────────┘ └──────────────┘ └──────────────┘                  │
│                                                                      │
│  → Pod가 여러 Node에 분산됨 (1대 고장 → 다른 Node가 대체)               │
└──────────────────────────────────────────────────────────────────────┘

ex08 프로젝트를 멀티 노드로 배포하면?

ex08의 4개 서비스가 3개 Worker Node에 분산되는 예시:

  Worker Node 1          Worker Node 2          Worker Node 3
  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
  │ Frontend Pod │      │ Backend Pod  │      │ Backend Pod  │
  │ Redis Pod    │      │ DB Pod       │      │              │
  │              │      │ [EBS 디스크]  │      │ (여유 공간)   │
  └──────────────┘      └──────────────┘      └──────────────┘
        │                     │                      │
        └─────────Service─────┘──────────────────────┘
              (자동 로드밸런싱)

  Backend replicas: 2이므로 → Node 2, Node 3에 1개씩 분산
  DB replicas: 1이므로 → Node 2에 1개 (데이터 일관성 때문)
  K8s Scheduler가 자동으로 "어느 Node에 배치할지" 결정한다
한 줄 요약: 노드 = 서버 1대. Minikube는 서버 1대에 전부 몰아넣는 것이고, 프로덕션은 여러 서버에 분산해서 죽어도 살아남게 만드는 것이다.

4. 필요 프로그램 설치 (Windows)

4-1. Docker Desktop

쿠버네티스의 컨테이너 런타임으로 Docker를 사용한다.

  1. Docker Desktop 공식 사이트에서 다운로드
  2. 설치 후 재부팅
  3. 확인:
docker --version

4-2. kubectl

# Chocolatey로 설치
choco install kubernetes-cli

# 또는 직접 다운로드
curl -LO "https://dl.k8s.io/release/v1.32.0/bin/windows/amd64/kubectl.exe"
# 다운로드한 파일을 PATH가 등록된 폴더에 이동

설치 확인:

kubectl version --client

4-3. Minikube

# Chocolatey로 설치
choco install minikube

# 또는 직접 다운로드
# https://minikube.sigs.k8s.io/docs/start/ 에서 Windows 인스톨러 다운로드

설치 확인:

minikube version

4-4. 전체 설치 확인

docker --version
kubectl version --client
minikube version

세 명령어가 모두 버전을 출력하면 준비 완료.

6. 실습 순서

Step 1. Minikube 시작

minikube start

로컬에 싱글 노드 쿠버네티스 클러스터를 생성한다. Docker Desktop이 실행 중이어야 한다.

Step 2. Docker 이미지 빌드

Minikube 내부에서 사용할 이미지를 빌드한다. docker build가 아니라 minikube image build를 사용해야 Minikube 클러스터 안에서 이미지를 인식한다.

minikube image build -t metacoding/db:1 ./db
minikube image build -t metacoding/backend:1 ./backend
minikube image build -t metacoding/frontend:1 ./frontend
minikube image build -t metacoding/redis:1 ./redis

Step 3. Namespace 생성

kubectl apply -f k8s/namespace.yml

metacoding 네임스페이스를 먼저 만든다. 다른 리소스들이 이 네임스페이스에 속하므로 가장 먼저 실행해야 한다.

Step 4. 모든 리소스 배포

kubectl apply -f k8s/ --recursive

--recursive 옵션으로 k8s/ 하위의 모든 YAML 파일을 한 번에 적용한다. Deployment, Service, ConfigMap, Secret, PV, PVC가 모두 생성된다.

Step 5. 배포 상태 확인

kubectl get deploy,pod,service -n metacoding

모든 Pod의 STATUS가 Running이 될 때까지 기다린다. Backend는 빌드 시간이 걸리므로 수 분 소요될 수 있다.

Step 6. 로그 확인 (문제 발생 시)

kubectl logs deploy/db-deploy -n metacoding --tail=100
kubectl logs deploy/backend-deploy -n metacoding --tail=100
kubectl logs deploy/frontend-deploy -n metacoding --tail=100

Step 7. 서비스 접속

minikube service frontend-service -n metacoding --url

출력된 URL을 브라우저에서 열면 사용자 리스트와 방문 횟수가 표시된다.

8. Minikube vs Kind vs kubeadm 비교

Docker Desktop과의 관계

Minikube는 기본적으로 자체 VM(가상머신)을 띄워서 그 안에 쿠버네티스를 실행한다. Docker Desktop도 내부적으로 VM을 사용하므로, 둘을 동시에 실행하면 VM끼리 리소스(CPU, 메모리)를 경쟁하게 된다.

반면 Kind와 kubeadm은 Docker 컨테이너 위에서 쿠버네티스 노드를 실행하므로, Docker Desktop과 자연스럽게 공존할 수 있다.

Docker Desktop (VM)
├── Docker Engine
│   ├── Kind      → Docker 컨테이너로 K8s 노드 실행 ✅ 공존
│   └── kubeadm   → Docker 컨테이너를 런타임으로 사용 ✅ 공존
│
Minikube (별도 VM) → Docker Desktop과 VM 경쟁 ⚠️ 충돌 가능

이미지 로드 방식 비교

로컬에서 빌드한 Docker 이미지를 클러스터 안에서 사용하려면 각 도구마다 방법이 다르다.

# Minikube — Minikube 내부에서 직접 빌드
minikube image build -t myapp:1 ./app
# 또는 로컬 이미지를 로드
minikube image load myapp:1

# Kind — 로컬 이미지를 클러스터에 로드
docker build -t myapp:1 ./app
kind load docker-image myapp:1

# kubeadm — Docker가 런타임이면 그냥 사용 가능
docker build -t myapp:1 ./app
# 별도 로드 불필요 (같은 Docker 데몬 사용 시)
# 레지스트리 사용 시: docker push → YAML에서 pull

Kind 멀티 노드 설정 예시

# kind-config.yml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
kind create cluster --config kind-config.yml
