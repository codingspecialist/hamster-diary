# 터널링 vs 포트포워딩 vs NodePort

## 핵심 차이 한눈에 보기

| 구분 | 터널링 | 포트포워딩 | NodePort |
|------|--------|-----------|----------|
| 비유 | 비밀 지하통로 | 안내데스크 전화 연결 | 건물 외벽에 뚫은 창구 |
| 목적 | 두 네트워크를 하나처럼 연결 | 외부 요청을 내부 특정 서비스로 연결 | 클러스터 외부에서 Pod 접근 |
| 암호화 | O (보통 있음) | X (단순 전달) | X (단순 전달) |
| 계층 | L3 (네트워크) | L4 (전송) | L4 (전송) |

---

## 1. 터널링 - "비밀 지하통로"

두 라우터가 미리 암호 약속(IKE)을 하고, 패킷을 암호화해서 겉에 새 주소를 붙여 보내는 방식.
중간에 누가 가로채도 내용을 볼 수 없다.

### 패킷 이동 과정

```
[서울 사무실]                    인터넷                     [부산 사무실]
192.168.10.0/24                                          192.168.80.0/24

PC-A                     라우터A       라우터B                    PC-B
192.168.10.5            3.3.3.1       9.9.9.1              192.168.80.20
    |                      |             |                      |
    |  src: 192.168.10.5   |             |                      |
    |  dst: 192.168.80.20  |             |                      |
    |--------------------->|             |                      |
    |                      |             |                      |
    |              [암호화 + 캡슐화]      |                      |
    |                      |             |                      |
    |            겉봉투:                 |                      |
    |            src: 3.3.3.1            |                      |
    |            dst: 9.9.9.1            |                      |
    |            내용: %%%암호화%%%       |                      |
    |                      |============>|                      |
    |                      |   터널구간   |                      |
    |                      |  (암호화됨)  |                      |
    |                      |             |                      |
    |                      |      [역캡슐화 + 복호화]            |
    |                      |             |                      |
    |                      |             |  src: 192.168.10.5   |
    |                      |             |  dst: 192.168.80.20  |
    |                      |             |--------------------->|
    |                      |             |                      |
```

**포인트**: 인터넷 구간에서는 `3.3.3.1 -> 9.9.9.1`만 보인다. 내부 IP(`192.168.x.x`)는 암호화되어 숨겨져 있다.

### 실제 예시: VPN으로 회사 네트워크 접속

```bash
# 집에서 회사 VPN 연결
# 내 PC(집): 172.30.1.50
# VPN 터널 생성 후 -> 회사 내부 IP 10.0.5.100 할당받음
# 이제 회사 DB서버 10.0.5.200에 직접 접근 가능

ssh user@10.0.5.200   # VPN 터널 덕분에 마치 회사 안에 있는 것처럼 접속
```

## 2. 포트포워딩 - "안내데스크 전화 연결"

외부에서 들어온 요청의 **포트 번호**를 보고, 내부의 특정 서비스로 연결해주는 방식.
암호화 없이 단순히 "이 포트로 오면 저쪽으로 보내줘"만 한다.

### 패킷 이동 과정

```
[외부]                        [공유기/라우터]                  [내부 네트워크]
                              공인 IP: 59.10.20.1            192.168.0.0/24

클라이언트                         NAT                    웹서버         DB서버
1.1.1.100                     59.10.20.1            192.168.0.10    192.168.0.20
    |                              |                     |               |
    |  dst: 59.10.20.1:8080        |                     |               |
    |---------------------------->|                     |               |
    |                              |                     |               |
    |                     [포트포워딩 규칙 확인]            |               |
    |                     8080 -> 192.168.0.10:80         |               |
    |                     3306 -> 192.168.0.20:3306       |               |
    |                              |                     |               |
    |                              | dst: 192.168.0.10:80|               |
    |                              |-------------------->|               |
    |                              |                     |               |
    |                              |    응답              |               |
    |                              |<--------------------|               |
    |  src: 59.10.20.1:8080        |                     |               |
    |<-----------------------------|                     |               |
```

**포인트**: 외부에서는 `59.10.20.1:8080`만 알면 된다. 내부에서 누가 처리하는지 모른다.

### 실제 예시: 공유기 포트포워딩 설정

```
공유기 포트포워딩 규칙:
  외부 포트 8080  -->  192.168.0.10:80   (웹서버)
  외부 포트 8443  -->  192.168.0.10:443  (웹서버 HTTPS)
  외부 포트 3306  -->  192.168.0.20:3306 (DB서버)
  외부 포트 2222  -->  192.168.0.30:22   (SSH서버)
```

### Docker에서의 포트포워딩

```yaml
# docker-compose.yml
services:
  nginx:
    image: nginx
    ports:
      - "8080:80"     # 호스트 8080 -> 컨테이너 80
```

```
[외부/호스트]                   [Docker 네트워크]

브라우저                     docker0 bridge              nginx 컨테이너
                            172.17.0.1                  172.17.0.2
    |                            |                          |
    |  dst: localhost:8080       |                          |
    |--------------------------->|                          |
    |                            |                          |
    |                   [iptables NAT 규칙]                 |
    |                   8080 -> 172.17.0.2:80               |
    |                            |                          |
    |                            | dst: 172.17.0.2:80       |
    |                            |------------------------->|
    |                            |                          |
    |                            |      HTTP 응답            |
    |                            |<-------------------------|
    |  src: localhost:8080       |                          |
    |<---------------------------|                          |
```

## 3. NodePort - "건물 외벽에 뚫은 창구"

Kubernetes에서 **모든 노드의 특정 포트**를 열어서, 외부에서 클러스터 내부 Pod에 접근할 수 있게 하는 방식.
본질은 **포트포워딩과 같은 개념**이다.

### 패킷 이동 과정

```
[외부]                    [K8s 클러스터]

                    Node1 (10.0.1.10)          Node2 (10.0.1.11)
                    NodePort: 30080            NodePort: 30080
                         |                          |
클라이언트               |    Pod-A                 |    Pod-B
1.1.1.100               |    10.244.1.5            |    10.244.2.8
    |                    |        |                 |        |
    |  dst: 10.0.1.10:30080      |                 |        |
    |-------------------->       |                 |        |
    |                    |       |                 |        |
    |           [kube-proxy 규칙 확인]              |        |
    |           30080 -> Pod-A(10.244.1.5:80)      |        |
    |                또는 Pod-B(10.244.2.8:80)      |        |
    |                    |       |                 |        |
    |                    | dst: 10.244.1.5:80      |        |
    |                    |------>|                 |        |
    |                    |       |                 |        |
    |                    |  응답  |                 |        |
    |                    |<------|                 |        |
    |  src: 10.0.1.10:30080      |                 |        |
    |<-------------------|       |                 |        |
```

**포인트**: 어느 노드로 접근하든(`Node1:30080`이든 `Node2:30080`이든) kube-proxy가 알아서 적절한 Pod로 연결해준다.

### 실제 예시: NodePort 서비스

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-web
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80          # 클러스터 내부에서 접근하는 포트 (ClusterIP:80)
      targetPort: 8080   # Pod이 실제 듣는 포트
      nodePort: 30080    # 외부에서 접근하는 포트 (모든 노드에 열림)
```

```
외부 접근:     http://<아무-노드-IP>:30080
                        |
                   [NodePort 30080]
                        |
                   [ClusterIP:80]        <-- 클러스터 내부 가상 IP
                        |
              +---------+---------+
              |                   |
         Pod-A:8080          Pod-B:8080   <-- 실제 컨테이너
```

## 비유 총정리

```
터널링 = 비밀 지하통로
========================================
  서울 사무실 =====[암호화된 터널]====== 부산 사무실
  마치 같은 건물인 것처럼 통신
  중간에 누가 봐도 내용 모름


포트포워딩 = 안내데스크 전화 연결
========================================
  외부 손님: "8080번으로 왔습니다"
  안내데스크: "아, 그건 3층 웹서버실로 연결해드릴게요"
  손님은 내부 구조를 몰라도 됨


NodePort = 건물 외벽에 뚫은 창구
========================================
  건물이 3개 있는 단지(클러스터)
  3개 건물 모두 30080번 창구가 있음
  어느 건물 창구로 가든 -> 내부 담당자(Pod)에게 전달됨

  결국 포트포워딩의 쿠버네티스 버전!
```

### NodePort는 포트포워딩인가?

**Yes.** NodePort의 본질은 포트포워딩이다.

| | 공유기 포트포워딩 | Docker -p | K8s NodePort |
|---|---|---|---|
| 누가 해줌 | 공유기(NAT) | iptables(Docker) | kube-proxy |
| 외부 포트 | 사용자 지정 | 사용자 지정 | 30000~32767 |
| 내부 목적지 | 내부 IP:포트 | 컨테이너 IP:포트 | Pod IP:포트 |
| 핵심 동작 | 포트 보고 전달 | 포트 보고 전달 | 포트 보고 전달 |

전부 "**외부 포트 X로 들어오면 -> 내부 Y로 보내줘**"라는 같은 원리다.
